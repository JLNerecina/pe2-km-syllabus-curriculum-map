## Entry 9
**Date:** 2026-05-01  
**Task:** Create separate developer and UI/UX sprint plans with role-specific deliverables and test case linkage.

**Prompt given to AI:**  
"Create a different sprint 1 and sprint 2 for developer and ui/ux but base it on the test cases, the completed ones is in sprint 1."

**What the AI produced:**  
Two comprehensive sprint documents:
- **Developer Sprints** — Sprint 1 (27 pts, all ✅ Done) with task breakdown per feature + all 8 passing test cases linked; Sprint 2 (26 pts) with 22 test cases in 4 modules (Student Dashboard, Faculty Dashboard, Admin, Super Admin)
- **UI/UX Sprints** — Sprint 1 (21 pts, all ✅ Done) with 5 screens delivered; Sprint 2 (23 pts) with 5 screens not started + design QA checklist per screen

**What I changed / improved:**  
I added velocity tracking tables and made the test case linkage explicit so QA and developers can coordinate. I also adjusted the task breakdowns to ensure non-blocking work between roles (UI delivered before dev starts implementing).

**Why:**  
The original sprint plan was agnostic to role. By splitting it, each team member has a crystal-clear roadmap of what they own and when they need to handoff to the next person. This prevents bottlenecks.

**What I learned / decided:**  
Parallel work is the key to meeting deadlines. Dev needs UI designs ready at sprint start, QA needs test cases written before dev finishes, and the KM Analyst needs to work independently. Decided to use this role-based sprint structure for all future sprint planning and to enforce handoff dates strictly via GitHub Projects milestones.

A complete user stories document with:
- Full product backlog (14 user stories, 64 story points) covering all roles

I adjusted the story point estimates based on complexity (e.g., prerequisite enforcement = 8 pts for heavy logic) and ensured each story mapped to at least one test case. I also made the velocity target realistic given the team's sprint capacity.
